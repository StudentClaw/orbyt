# Architecture Layer: External Services (Tier 3)

## What It Is

Tier 3 encompasses everything outside of Student Claw that the app communicates with: the Codex CLI subprocess (which connects to OpenAI), the Canvas LMS REST API, and any MCP server plugins that reach external services (Google Calendar, Apple Calendar, Notion, etc.). The Local Server manages all external communication — nothing reaches the React UI directly.

---

## Why It's a Separate Layer

External services are unreliable. APIs go down, tokens expire, rate limits kick in, and latency varies wildly. By isolating all external communication in Tier 3 (behind the Local Server), the rest of the app can handle failures gracefully:

- The UI never hangs waiting for Canvas
- A crashed MCP plugin doesn't affect the chat
- Cached data keeps the Dashboard useful even when external services are unavailable

---

## External Service Inventory

### 1. Codex CLI (AI Subprocess)

**What**: OpenAI's Codex CLI, running as a child process of the Local Server.
**Protocol**: JSON-RPC 2.0 over stdin/stdout.
**Auth**: ChatGPT subscription (browser OAuth) or OpenAI API key.
**Failure modes**: Process crash, timeout, auth expiry, rate limiting, network loss.

Detailed in: [01-ai-harness.md](../features/01-ai-harness.md)

### 2. Canvas LMS REST API

**What**: The university's Canvas instance, accessed via REST API with a personal access token.
**Protocol**: HTTPS REST, JSON responses, Link-header pagination.
**Auth**: Personal access token (generated by student, stored in Vault).
**Failure modes**: Token expiry, rate limiting (Canvas returns 403), server downtime, pagination edge cases.
**Base URL**: Per-university (e.g., `https://canvas.myuniversity.edu/api/v1/`).

Detailed in: [02-canvas-integration.md](../features/02-canvas-integration.md)

### 3. Calendar Services (via MCP)

**What**: Apple Calendar or Google Calendar, accessed through an MCP server plugin.
**Protocol**: MCP over stdio (plugin runs in utilityProcess).
**Auth**: Google OAuth (for Google Calendar) or local EventKit access (for Apple Calendar).
**Tools exposed**: `get_free_blocks`, `create_event`, `update_event`, `delete_event`.
**Failure modes**: OAuth token expiry, calendar API rate limits, permission denied.

### 4. Notion (via MCP)

**What**: Notion workspace access for students who keep notes there.
**Protocol**: MCP over stdio.
**Auth**: Notion integration token.
**Tools exposed**: `search_pages`, `read_page`, `create_page`, `append_block`.
**MCP availability**: Official Notion MCP server exists on MCP Market.

### 5. Future Services

The plugin architecture means any service with an MCP server (28,000+ available) can be integrated:

| Service | Use Case | MCP Status |
|---|---|---|
| GitHub | CS students: repos, issues, PRs | Official MCP exists |
| Google Docs/Sheets | Document editing, spreadsheet data | Official MCPs exist |

## Communication Patterns

### Pattern 1: Request/Response (Canvas API, Calendar)

```
Server sends request → External service returns response
Timeout: 10-30 seconds depending on service
Retry: Up to 3 times with exponential backoff for transient errors
Cache: Response cached in SQLite for offline resilience
```

### Pattern 2: Long-Running Stream (Codex CLI)

```
Server sends prompt → Codex CLI streams tokens over seconds/minutes
Interruptible: User can cancel mid-stream
Tool calls: Stream pauses, tool executes, stream resumes
Timeout: Configurable per-request (default: 2 minutes)
```

### Pattern 3: Background Sync (Canvas)

```
Scheduled trigger (every 30 min) → Fetch changes since last sync
Diff against local cache → Update SQLite → Push changes to UI via WS
Non-blocking: Runs in a background Effect Fiber
Smart: Only fetches what changed (If-Modified-Since, updated_at filters)
```

## Resilience Strategy

Every external call in Student Claw goes through the same resilience pipeline (Effect-TS):

```
External Call
  ├── Timeout (configurable per service)
  ├── Retry (exponential backoff, max 3 attempts for transient errors)
  ├── Circuit Breaker (stop calling a service that's been failing)
  ├── Fallback (serve cached data from SQLite)
  └── Error Typing (all failures tracked in the Effect error channel)
```

**Offline mode**: When a student has no internet:
- Dashboard shows cached data with a "Last synced: X minutes ago" indicator
- Chat works if Codex CLI is already authenticated (session cached)
- File viewer, skills, and memory all work fully offline (they're local)
- Canvas sync pauses and resumes when connectivity returns

## Security Boundaries

```
React UI (Renderer)
  │
  │ ✗ Cannot make network requests
  │ ✗ Cannot access filesystem
  │ ✗ Cannot spawn processes
  │
  ▼
Local Server (Child Process)
  │
  │ ✓ Makes all network requests
  │ ✓ Manages all file I/O
  │ ✓ Spawns Codex CLI
  │
  ▼
MCP Plugins (utilityProcess)
  │
  │ ✓ Make requests to their specific service (Canvas, Calendar, etc.)
  │ ✗ Cannot access other plugins' credentials
  │ ✗ Cannot access the UI
  │ ✗ Cannot access the filesystem outside their sandbox
```

## Proposed File Structure

External services don't have their own package — they're accessed through the server and plugins:

```
packages/server/src/
  ai/
    CodexCli.ts                 # Manages Codex CLI subprocess
  mcp/
    Orchestrator.ts             # Manages MCP plugin communication
    ProtocolAdapter.ts          # MCP protocol encoding/decoding
    ToolRouter.ts               # Route tool calls to the right plugin

packages/extensions/
  canvas-mcp/                   # Canvas LMS MCP server
  calendar-mcp/                 # Calendar service MCP server
  notion-mcp/                   # Notion MCP server
  template-mcp/                 # Template for new plugins
```

## Open Questions

- **Rate limiting coordination**: When multiple features hit Canvas simultaneously (background sync + user query), how do we share rate limit budgets?
- **Credential rotation**: When tokens expire, the flow is: detect failure → notify user → guide re-auth → retry. How seamless can we make this?
- **MCP transport**: Local plugins use stdio. Should we support SSE for future remote MCP servers?
- **Service health dashboard**: Should the Extension Manager show the health status of each external connection (green/yellow/red)?
- **Analytics**: Should we track (locally) how often each service is called, response times, failure rates? Useful for debugging but adds complexity.
